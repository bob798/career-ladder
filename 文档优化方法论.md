# 文档优化方法论 · 长文去 AI 味与结构化

> 用途：把一篇内容扎实但"AI 味重、堆叠、概念散"的长文知识文档，系统优化成清爽、可读、可视化的版本。
> 适用：本仓库的《软件工程师之路》《产品经理之路》《运营之路》等同构长文，以及任何中文长文知识文档。
> 这套流程在《软件工程师之路》上验证有效（加粗 166→4、全角破折号 54→13），可直接复用。

---

## 五个优化维度（要解决什么）

| 维度 | 典型病征 |
|---|---|
| 去 AI 味 | 加粗泛滥、破折号当万能连接、表演性动词（钉死/驳掉/说破/点透）、emoji ⚠️ |
| 内容结构化 | 同一信息出现多遍、强对仗模板（如"七件套"）堆叠 |
| 概念统一 | 同义异名（影响力半径 vs 问题半径）、记号顺序不一（IC/L 谁前谁后） |
| 样式优化 | 引用块 `>` 语义混乱、导读过长、箭头滥用 |
| 增加可视化 | 纯 ASCII 图、映射散落全文没有收口图 |

---

## 七步流程（怎么做，按顺序执行）

### 1. 先建度量，再动手
把"AI 味""堆叠"这类主观词翻译成**可数指标**，改完用 `grep` 对比验证，用证据而非感觉。

```bash
grep -o '\*\*' 文件.md | wc -l        # 加粗数量
grep -o '——' 文件.md | wc -l          # 全角破折号数量
grep -c '```' 文件.md                  # 代码/图表围栏数
```

### 2. 去重靠「单一信息源」
每条信息只在一处完整出现，别处引用。例：技术底盘归一节、可观察证据归自测工具节、暗线归一节——其它地方只点名引用，不重述。

### 3. 概念统一靠「主锚 + 别名收敛」
选一个贯穿全文的主线词（如"影响力半径"或"信任边界"），把所有同义异名收敛过去；职级记号（IC/L）统一一种顺序。改完 grep 搜旧名确认零残留。

### 4. 去 AI 味靠「删强调 + 平实动词」
加粗、破折号、表演性动词是三大来源。
- **判据**：去掉加粗后句子仍成立，就删——一段最多留 1 个加粗。
- 破折号大部分换成句号或冒号，一段留 1 个。
- 表演性动词换平实词：钉死→定下、驳掉→反驳、说破→说清、点透→点出。
- 删 emoji（中文技术文里 ⚠️ 最显 AI）。

### 5. 结构化靠「模板降维」
强对仗模板（如每级"七件套"）压成"表格 + 一段散文"，信息不丢、堆叠感消失。

### 6. 可视化靠「原生渲染 + 收口图」
- ASCII 图 → Mermaid（GitHub 原生渲染）。
- 散落全文的映射用**一张总览矩阵**收口（如"5 级 × 4 条线各到哪"）。
- 别图表泛滥：简单线性流用行内 `→` 文字即可，不必都转 Mermaid，且同一文档内可视化风格要统一。

### 7. 回扫验证（必做）
用同一批 grep 指标回扫，既确认达标也抓残留。实测能抓到术语漏网、重复图、风格割裂等改写时遗漏的问题。

```bash
grep -n "旧术语" 文件.md          # 概念统一是否干净
grep -nE '^## [0-9]' 文件.md      # 章节锚点 §N 是否都存在
grep -o '\*\*' 文件.md | wc -l    # 加粗是否回落到目标区间
```

## 验收基线（来自《软件工程师之路》）

| 指标 | 改前 | 改后 |
|---|---|---|
| 加粗 `**…**` | 166 处 | 4 处 |
| 全角破折号 `——` | 54 | 13 |
| Mermaid 图 | 0 | 2 |
| emoji ⚠️ | 1 | 0 |
| 重复信息块 | 3 处 | 0 |

> 一句话总结：不靠手感改文风，而是**把主观感受量化成 grep 指标 → 分维度改 → 回扫验证**。先建度量、最后回扫，是这套方法论里最容易被跳过、却最值钱的两步。
